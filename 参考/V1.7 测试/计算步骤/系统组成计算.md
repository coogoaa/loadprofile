## 系统组成计算

### 计算目标

基于三套方案（A/B/C）的规则，根据屋顶初始PV容量确定各方案的目标系统容量，选择最优面板组合，并与API返回结果进行比对验证。

### 系统类型判断

在计算之前，首先根据 `detect_building.json` 中的 `data.with_pv` 字段判断系统类型：

| with_pv | 系统类型 | 说明 |
|---------|---------|------|
| false | **新建系统** | 全新安装光伏+储能系统，屋顶按理论最大容量计算 |
| true | **储能扩容** | 已有光伏系统，仅加装储能，假设屋顶已使用 60% 容量 |

> ⚠️ `detect_building.json` 缺失时，默认按新建系统处理，并在报告中给出警告。

两种系统类型的核心差异：

| 差异项 | 新建系统 | 储能扩容 |
|--------|---------|---------|
| 初始面板数 | 物理最大（铺满） | FLOOR(物理最大 × 0.6) |
| 方案A面板 | 铺满，容配比校验后可能削减 | 固定，不可削减 |
| 方案B/C面板 | 按目标容量削减 | 固定，不可削减（三套方案共用同一组面板） |
| 容配比超标处理 | 先增大逆变器，再削减面板 | 只能增大逆变器，不能削减面板（超标则触发兜底） |
| 光伏剩余系数A | 0.7 | 0.6 |
| 光伏剩余系数B | 0.55 | 0.45 |
| 报价公式 | 面板+逆变器+电池 | 逆变器+电池（不含面板） |
| API比对方式 | 面板列表直接比对 | 面板列表不比对（API返回全部面板），通过cashflow验证 |

### 三套方案对应关系

| 方案 | 标签 | 目标 | 策略 |
|------|------|------|------|
| 方案A | maxValue | 物理极限满铺 | 铺满，容配比校验后可能削减 |
| 方案B | mostPopular | 市场主流容量 | >15kW→13.3kW, ≤15kW→10.12kW |
| 方案C | customFit | 入门容量 | 6.6kW |

### 预设参数

```
单块面板功率: 0.44 kW (440W)
目标容配比: 180%
最大容配比: 200%
单相逆变器最大功率: 10 kW

逆变器规格:
  方案A: [5, 6, 8, 10] kW
  方案B: [5, 6, 8] kW
  方案C: [5, 6, 8] kW

电池效率: DoD 90% × RTE 95% = 85.5%
电池规格库: [5, 6.5, 9.6, 10, 13.5, 16, 20, 25, 30, 35, 40, 45, 50] kWh
电池容量限制: 最小 5 kWh，最大 50 kWh

光伏剩余系数（新建系统）:
  方案A: 0.7
  方案B: 0.55

光伏剩余系数（储能扩容）:
  方案A: 0.6
  方案B: 0.45

储能扩容参数:
  屋顶容量使用系数: 0.6

成本参数:
  面板: 540 AUD/kW
  逆变器: 280 AUD/kW
  电池: 865 AUD/kWh
  GST: 10%
```

### 计算步骤

#### 0. 系统类型判断

```python
读取 detect_building.json → data.with_pv
if with_pv == True:
    系统类型 = 储能扩容
else:
    系统类型 = 新建系统
```

#### 1. 数据准备

需要以下数据文件（参考 `相关变量数据.md`）：

- **request.json**: 计算请求参数，包含项目基本信息
- **panel_location.json**: preHandler处理后的面板位置和发电量数据
- **detect_building.json**: GIS楼宇检测结果（含 `with_pv` 字段）
- **maxValue_panel_location.json**: 方案A的API返回面板数据
- **mostPopular_panel_location.json**: 方案B的API返回面板数据
- **customFit_panel_location.json**: 方案C的API返回面板数据
- **maxValue_cashflow.json**: 方案A的现金流数据
- **mostPopular_cashflow.json**: 方案B的现金流数据
- **customFit_cashflow.json**: 方案C的现金流数据

#### 2. 计算初始PV容量

```python
初始PV容量 (kW) = 面板数量 × 单块面板功率

其中：
- 面板数量：从 panel_location.json 中统计
- 单块面板功率：0.44 kW (440W)
```

**示例**（项目5794，新建系统）：
- 12块面板
- 初始容量 = 12 × 0.44 = 5.28 kW

**示例**（项目5810，储能扩容）：
- 17块面板（物理最大）
- 储能扩容面板 = FLOOR(17 × 0.6) = 10 块
- 储能扩容容量 = 10 × 0.44 = 4.40 kW（固定，不可削减）

#### 3. 屋顶坡面分析

```python
1. 按 roofId 对面板分组
2. 统计每个坡面的面板数、方位角、总发电量、平均发电量
3. 按平均发电量排序（用于后续削减策略）
```

#### 4. 用电需求计算

```python
年用电量 = 各州预设值（如 VIC = 6778 kWh）
日均用电 = 年用电量 / 365
白天用电 (07:00-17:00) = 日均 × 白天小时比例之和 / 100
晚高峰用电 (17:00-21:00) = 日均 × 晚高峰小时比例之和 / 100
整夜用电 (17:00-07:00) = 日均 × 整夜小时比例之和 / 100
```

### 新建系统计算（with_pv = false）

#### 5. 方案A (maxValue) 计算

```python
1. 目标: 物理极限满铺，所有面板全选
2. 逆变器初选: 选择 ≥ (PV容量 / 1.8) 的最小规格
3. 容配比校验:
   - 若 ≤ 200%: 通过
   - 若 > 200%: 尝试增大逆变器，仍超则从发电量最小的坡面开始削减面板
4. 电池容量:
   - 整夜需求 = 整夜用电
   - 光伏剩余电池 = 0.7 × max(0, 日均发电 - 白天用电)
   - E_req = max(整夜需求, 光伏剩余电池)
   - Battery_calc = E_req / 0.855
   - Battery_nominal = 向上取整到规格库
```

#### 6. 方案B (mostPopular) 计算

```python
1. 确定目标容量:
   - 初始容量 > 15kW → 目标 13.3kW
   - 初始容量 ≤ 15kW → 目标 10.12kW
2. 判断容量是否不足:
   - 若初始容量 ≤ 目标容量: 铺满（全选）
   - 若初始容量 > 目标容量: 从发电量最小的坡面开始削减
3. 面板削减: CEILING(目标容量 / 0.44) 块
4. 逆变器选型 + 容配比校验（同方案A）
5. 电池容量:
   - 晚高峰需求 = 晚高峰用电
   - 光伏剩余电池 = 0.55 × max(0, 日均发电 - 白天用电)
   - E_req = max(晚高峰需求, 光伏剩余电池)
   - Battery_calc = E_req / 0.855
   - Battery_nominal = 向上取整到规格库
```

#### 7. 方案C (customFit) 计算

```python
1. 目标容量: 6.6kW
2. 判断容量是否不足:
   - 若初始容量 ≤ 6.6kW: 铺满（全选）
   - 若初始容量 > 6.6kW: 从发电量最小的坡面开始削减
3. 面板削减: CEILING(6.6 / 0.44) = 15 块
4. 逆变器选型 + 容配比校验（同方案A）
5. 电池容量:
   - E_req = 晚高峰用电
   - Battery_calc = E_req / 0.855
   - Battery_nominal = 向上取整到规格库
```

#### 8. 面板削减策略

```python
削减规则（适用于新建系统方案A容配比超标、方案B/C需要削减时）:
1. 按坡面平均发电量升序排列所有坡面
2. 从发电量最小的坡面开始，逐块删除发电量最小的面板
3. 直到面板数 ≤ 目标面板数
```

#### 9. 新建系统 - 计算结果与API返回结果比对

对每套方案，比对以下指标：

```python
1. 面板数量: 计算选出的面板数 vs API返回的面板数
2. 面板列表: 用positions坐标作为唯一标识，比对面板集合
3. PV容量: 计算容量 vs API面板数 × 0.44
4. 年发电量: 计算发电量 vs API面板发电量之和
```

### 储能扩容计算（with_pv = true）

#### 5E. 储能扩容面板选择（三套方案共用）

```python
1. 计算扩容面板数: FLOOR(物理最大面板数 × 0.6)
2. 从发电量最小的坡面开始削减，直到剩余面板数 = 扩容面板数
3. 扩容容量 = 扩容面板数 × 0.44 kW（固定，不可削减）
4. 三套方案（A/B/C）共用同一组面板
```

**示例**（项目5810，17块面板，QLD）：
- 扩容面板 = FLOOR(17 × 0.6) = 10 块
- 扩容容量 = 10 × 0.44 = 4.40 kW

#### 6E. 储能扩容逆变器选型

```python
1. 目标功率 = 扩容容量 / 1.8
2. 选择 ≥ 目标功率的最小逆变器规格
3. 容配比校验:
   - 若 ≤ 200%: 通过
   - 若 > 200%: 只能增大逆变器（不能削减面板！）
   - 若已达最大逆变器仍超标: 触发兜底卡片
```

> ⚠️ **关键区别**: 储能扩容不能削减面板，只能增大逆变器。若容配比仍超 200% 则方案不可行，触发兜底。

#### 7E. 储能扩容电池容量计算

```python
方案A:
  E_req = max(整夜需求, 0.60 × 光伏剩余)    # 注意系数 0.60（非新建的 0.7）
  Battery_calc = E_req / 0.855
  Battery_nominal = 向上取整到规格库

方案B:
  E_req = max(晚高峰需求, 0.45 × 光伏剩余)  # 注意系数 0.45（非新建的 0.55）
  Battery_calc = E_req / 0.855
  Battery_nominal = 向上取整到规格库

方案C:
  E_req = 晚高峰需求
  Battery_calc = E_req / 0.855
  Battery_nominal = 向上取整到规格库
```

> **注意**: 当光伏剩余较小时（小屋顶或扩容后面板少），方案B的 `0.45 × 光伏剩余` 可能小于晚高峰需求，
> 此时 B 和 C 的 E_req 都取晚高峰需求，导致电池容量相同。这是正常的计算结果。
>
> **示例**（项目5810）：
> - 晚高峰需求 = 4.70 kWh
> - 光伏剩余 = 7.75 kWh
> - 方案B: E_req = max(4.70, 0.45 × 7.75) = max(4.70, 3.49) = **4.70** → 6.5 kWh
> - 方案C: E_req = **4.70** → 6.5 kWh
> - B 和 C 电池相同，因为光伏剩余项未起决定作用

#### 8E. 储能扩容报价计算

```python
税前报价 = 逆变器功率 × 280 + 电池容量 × 865   # 不含面板成本
含税报价 = 税前报价 × 1.1
最终报价 = 含税报价 - 补贴
```

> 储能扩容不含面板成本（面板已有），只计算逆变器和电池。

#### 9E. 储能扩容 - 计算结果与API返回结果比对

```python
储能扩容时，API返回的 {scheme}_panel_location.json 是屋顶全部面板（用于发电量参考），
不是储能扩容实际使用的面板子集。因此面板列表比对不适用。

验证方式:
1. 通过 cashflow 初始投资反推逆变器和电池配置
2. 计算含税报价 vs API cashflow[0] 的绝对值
```

### 验证要点

1. **系统类型判断**：确认 `detect_building.json` 中 `with_pv` 字段正确读取
2. **三套方案完整性**：确认A/B/C三套方案都已计算
3. **容量规则正确性**：检查各方案目标容量是否符合规则
4. **面板削减策略**：
   - 新建系统：验证是否从发电量最小的坡面开始削减
   - 储能扩容：验证面板固定不可削减，三套方案共用同一组面板
5. **逆变器选型**：验证容配比是否在 ≤200% 范围内
6. **电池容量**：验证电池计算公式和规格标准化，注意新建/扩容使用不同光伏剩余系数
7. **API比对**：
   - 新建系统：计算结果与 `{scheme}_panel_location.json` 的面板数量、列表、容量、发电量是否一致
   - 储能扩容：通过 cashflow 初始投资验证逆变器和电池配置
8. **特殊情况**：
   - 屋顶容量不足时，三套方案应全部铺满（面板列表相同）
   - 储能扩容小屋顶时，方案B和C电池可能相同（光伏剩余项未起决定作用）
   - 储能扩容大屋顶（>75块面板）可能触发兜底
