# Q&A：疑问解答与方案推荐改动分析

> 基于《新计算规则整理》文档的补充说明，覆盖四个问题：时段符号含义、NT无供暖、文档笔误说明、V1.5.0方案推荐策略改动分析。

---

## Q1：（H07-H16）（H17-H20）（H17-H06）表示什么？

这是 **24小时时间段的索引范围**，H 代表 Hour，数字是小时编号（0点=H00，23点=H23）。

| 符号 | 对应时间 | 含义 | 小时数 |
|------|---------|------|-------|
| H07–H16 | 早7点 → 下午5点前 | 白天用电时段 | 10小时 |
| H17–H20 | 下午5点 → 晚9点前 | 晚高峰用电时段 | 4小时 |
| H17–H06 | 下午5点 → 第二天早7点前 | 整夜用电时段（跨越午夜） | 14小时 |

**整夜时段的写法说明**：H17–H06 跨越午夜，实际是 H17–H23 加上 H00–H06 两段合并，简写为 H17–H06。

这三个时段直接对应 V1.5.0 方案推荐的三条电池容量推荐路径：
- **方案 A**（最大化）：覆盖整夜用电 → 依赖 H17–H06
- **方案 B**（主流）：晚高峰 + 储光伏剩余 → 依赖 H17–H20
- **方案 C**（入门）：只覆盖晚高峰 → 依赖 H17–H20

## Q2：—（无供暖）是什么意思？怎么得来的？

数据来源是 `参考/docs/presets/AU_cooling_heating_season.csv`，NT（北领地）那行原文：

```
NT, Sep-Apr, -, 热带气候;几乎全年需要制冷;无供暖需求
```

**含义**：北领地是热带气候，全年气温高，冬天（6–8月）气温仍在 20°C 以上，**不存在供暖需求**，所以暖季字段填 `—`（不适用/无）。

**对计算的影响**：
- NT 用户选 Heat pump 时，ThermalWeight 的暖季部分 = 0，热力额外电量**全部分配给制冷季**（Sep–Apr）
- NT 用户选 Electric heating 实际场景几乎不存在，产品层面可考虑在 NT 地区**隐藏 Electric heating 选项**

## Q3：文档笔误说明

原文档中有这样一行：

> `NormBase[May] = 572.0 / 8384.2 × 7778 = 530.6 kWh（之前写错了，取Sep值）`

**问题所在**：572.0 这个 ReshapedBase 数值不是 May 的，是 **Sep** 的（Sep 的 BaseShare = 0.07356，× 7778 × 1.0 = 572.0）。May 的 BaseShare = 0.08466，ReshapedBase 应为 658.6，NormBase 也对应不同。

括号里的注释"之前写错了，取Sep值"是写作时留下的自注，不应出现在正式文档中，**已标记为待修正的笔误**，不影响计算逻辑本身的正确性。

## Q4：V1.5.0 方案推荐策略需要改吗？

### 结论先行

- **方案推荐的公式结构：不改**
- **输入数据来源：必须替换**
- **投资回报自用率：需要适配逻辑**

---

### 层1：方案推荐公式——不改

V1.5.0 三套方案的核心推导逻辑不变：

```
光伏剩余 = 日均发电量 - 白天用电(H07-H16)

方案 A（最大化/全覆盖）：
  电池容量目标 = 整夜用电(H17-H06)

方案 B（主流/晚高峰+储余）：
  电池容量目标 = 晚高峰用电(H17-H20) + 光伏剩余储存

方案 C（入门）：
  电池容量目标 = 晚高峰用电(H17-H20)
```

公式骨架完全不动。

---

### 层2：输入数据来源——必须替换

| 数据项 | V1.5.0 来源 | V1.12 来源 | 改动性质 |
|--------|-----------|-----------|--------|
| 日均用电 | 州预设年电量 ÷ 365 | `Final_Annual_kWh ÷ 365` | 数据源替换 |
| 白天用电 | 日均 × **固定 43.88%** | 日均 × `SUM(FinalHourlyShare[H07..H16])` | 固定 → 动态 |
| 晚高峰用电 | 日均 × **固定 22.1%** | 日均 × `SUM(FinalHourlyShare[H17..H20])` | 固定 → 动态 |
| 整夜用电 | 日均 × **固定 59.6%** | 日均 × `SUM(FinalHourlyShare[H17..H23] + H00..H06)` | 固定 → 动态 |

**向后兼容保证**：当用户全部使用默认因子（Mostly away + No system + No EV）时，LoadProfile 输出的四个数字与旧版固定预设基本一致，不会破坏现有逻辑。

---

### 层3：投资回报自用率——需要适配

这是最需要真正改逻辑的地方。

**V1.5.0 自用率（用固定占比近似）：**
```
自用率 ≈ MIN(日均发电, 白天用电) / 日均发电
        ← 白天用电用固定 43.88% 算出
```

**V1.12 自用率（用动态24h分布精确计算）：**
```
无电池：
  自用率 = MIN(日均发电, 动态白天用电) / 日均发电

有电池：
  夜间电池放电量 = MIN(电池容量, 整夜用电)
  自用率 = (白天直接自用 + 夜间电池放电) / 日均发电
```

**典型场景下自用率变化幅度**（假设日均发电 15 kWh，NSW 基准）：

| 场景 | 白天用电 | 自用率 | 与旧版差值 |
|------|--------|-------|---------|
| 旧版固定（所有NSW） | 9.35 kWh | 62.3% | 基准 |
| WFH（居家办公） | ~10.7 kWh | 71.3% | **+9%** |
| Mostly away + EV 夜间充电 | ~7.1 kWh | 47.3% | **-15%** |
| WFH + EV 太阳能充电 | ~14.3 kWh | 95.3% | **+33%** |

自用率最大相差 **33 个百分点**，直接影响"年节省电费"和"回本周期"的准确性。不改的话，给不同用电习惯的用户展示的财务数字会严重失真。
